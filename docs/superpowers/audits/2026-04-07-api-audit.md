## 1. Resumen Ejecutivo

El proyecto cuenta con **36 endpoints API** distribuidos en `app/routes.php`. La infraestructura base existe (`AbstractApiController`, `ApiAuthMiddleware`, `ApiRoleMiddleware`, `ExceptionHandler` con detección API) pero la **coherencia de respuestas está fragmentada en al menos 5 formatos distintos** y hay **3 controladores PSR-15 incompatibles** que usan `header()` + `echo` + `exit` directamente. El OpenAPI spec documenta solo 10 de los 36 endpoints (28%) y usa prefijos de ruta inconsistentes con el código real.

### Veredicto de producción

| Área | Estado | Gravedad |
|------|--------|----------|
| Formato de respuesta | ❌ 5 formatos distintos coexisten | CRÍTICA |

## 3. Análisis de Formatos de Respuesta

### 3.1 Catálogo de Formatos Encontrados

Se han identificado **5 formatos distintos** de respuesta activos en producción:

#### Formato A — Canónico (AbstractApiController) ✅

```json
// Éxito:
{"ok": true, "data": <payload>}
// Éxito colección:
{"ok": true, "data": {"items": [...], "total": N, "meta": {...}}}
// Error:
{"ok": false, "error": "mensaje", "code": "error_code"}
```

**Origen:** `AbstractApiController::success()` / `error()` / `collection()`
**Usado por:** MenuController, LoyaltyController (parcial), TimeSlotController, WaitlistController

#### Formato B — Legacy "success" ❌

```json
{"success": true, "data": {...}}
{"success": false, "error": "mensaje"}
```

**Origen:** controladores con `$this->response->json(['success' => true, ...])` directo
**Usado por:** ReservationController, ManagerController, SupervisorController

#### Formato C — JsonResponse (PSR incompatible) ❌

```json
{"ok": true, "data": {...}}
{"ok": false, "error": "mensaje", "code": "error_code"}
```

**Origen:** `App\Core\JsonResponse::ok()` / `fail()` — llama `header()` + `echo` + `exit`
**Usado por:** CartController, FavoriteController, CookieController
**Problema crítico:** devuelven `void` (retorno nulo), no `ResponseInterface` PSR-15.
El middleware stack nunca recibe una respuesta; PHP termina la ejecución con `exit`.

#### Formato D — Closures inline ❌

```json
{"success": true}       // POST /api/cookies/save-preferences
{"success": true}       // POST /api/cookies/newsletter-prompted
{"prompted": false}     // GET  /api/cookies/newsletter-prompted
{"ok": true, "data": {...}}  // GET /api/cart/guest (semi-correcto)
```

**Origen:** closures en `routes.php` líneas ~100-140
**Problema:** lógica de negocio en archivo de rutas, no testable unitariamente

#### Formato E — Raw ResponseFactory (sin envelope) ⚠️

```json
// HolidayController@getHolidays:
{"holidays": [...], "count": 10}
// HolidayController@checkHoliday:
{"date": "...", "is_holiday": true, "available_for_reservations": false}
```

**Origen:** `$this->response->json([...])` sin envelope `{ok, data}`
**Usado por:** HolidayController, NewsletterApiController (éxito con `{ok:true, message:...}`)

## 4. Bugs y Anomalías Críticas

### BUG-001: Ruta `/loyalty/redeem` fuera del prefijo `/api/`

**Localización:** `app/routes.php` — la ruta `POST /loyalty/redeem` está registrada fuera del grupo con prefijo `/api/`.
**Impacto:** ExceptionHandler no detecta esta ruta como API request vía `str_starts_with($uri, '/api/')`. En caso de excepción, se devuelve una respuesta HTML en lugar de JSON.
**Solución:** mover al grupo `/api/v1/loyalty/`.

### BUG-002: LoyaltyController accede a `$_SESSION` directamente

**Localización:** `app/Http/Controllers/Api/LoyaltyController.php` — método `redeem()` usa `$_SESSION['user_id']` en lugar de `Session::userId()`.
**Impacto:** acopamiento al superglobal; si el motor de sesión cambia o se desactiva, falla silenciosamente.
**Solución:** sustituir por `Session::userId()`.

### BUG-003: CartController, FavoriteController y CookieController son PSR-15 incompatibles

**Localización:** `app/Core/JsonResponse.php` — método `send()` llama `header()` + `echo json_encode()` + termina con `exit`.
Los tres controladores declaran retorno `void` e invocan este helper.
**Impacto:** el middleware PSR-15 downstream nunca ejecuta. Si algún middleware de post-proceso (logging, CORS headers, rate limiting) se añade en el futuro, no se ejecutará para estos endpoints.
**Solución:** refactorizar los tres controladores para extender `AbstractApiController` o usar `ResponseFactory::json()` retornando `ResponseInterface`.

### BUG-004: Detección API no cubre rutas fuera de `/api/`

**Localización:** `app/Core/ExceptionHandler.php::isApiRequest()` — usa `str_starts_with($uri, '/api/')`.
**Impacto:** rutas fuera del prefijo (BUG-001) reciben respuesta HTML en errores.
**Solución:** corregir BUG-001 y/o añadir detección por `Accept: application/json`.

### BUG-005: CookieController tiene métodos con nombres no mapeados a rutas

**Localización:** `routes.php` registra rutas a `@accept`, `@reject`, `@update` pero la clase tiene métodos `acceptAll()`, `rejectOptional()`, `savePreferences()`.
**Impacto:** llamar `GET /api/cookies/accept` resultará en un error de método no encontrado.
**Verificar en ejecución con:** `GET /api/cookies/accept`

### BUG-006: Closures inline con formato incorrecto

**Localización:** `routes.php` ~líneas 110-140 — 4 rutas usan closures con `$responseFactory->json(['success' => true])`.
El formato `success` no es el formato canónico.
**Impacto:** inconsistencia para clientes ApiFirst; no testable unitariamente.

## 6. Estado de ExceptionHandler

El método `isApiRequest()` en `ExceptionHandler.php` detecta peticiones API via:

```php
str_contains($accept, 'application/json')        // Accept header
|| str_contains($contentType, 'application/json') // Content-Type header
|| strtolower($requestedWith) === 'xmlhttprequest'// X-Requested-With
|| str_starts_with($uri, '/api/')                 // URI prefix
```

**Evaluación:** correcto para el 97% de los casos. El único punto ciego es `/loyalty/redeem` (BUG-001) que no empieza por `/api/`.

El handler cubre **10 tipos de excepción** con ramas API y HTML separadas:

- ValidationException → 422 con `toArray()`
- NotFoundException → 404
- AuthenticationException → 401
- AuthorizationException → 403
- BusinessRuleException → 422
- RateLimitException → 429
- RouterParameterException / RouterException → 400/404
- MiddlewareException → 403
- DatabaseException → 500 (mensaje sanitizado en prod)
- ConfigurationException → 500
- ExternalServiceException → 502
- Default (generic) → 500
